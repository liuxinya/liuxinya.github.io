前后端开发模式

  1. 前后端混合开发
    前端：负责写出静态页面，所有跟数据相关的事情全都不用管
    后端：负责写后台业务逻辑，拿到前端静态页面之后，和后台代码进行整合，实现所有的数据交互

  2. 前后端半混合开发
    前端：负责写出静态页面，负责部分数据相关的内容
    后端：负责写后台业务逻辑，拿到前端静态页面之后，和后台代码进行整合，实现所有的数据交互

  3. 前后端完全分离开发
    前端：负责写出静态页面，负责页面中全部的数据交互内容（ajax）
    后端：只负责提供接口

浏览器和服务器通信过程

  1. 浏览器向服务器发送请求
  2. 服务器接收到请求之后，解析请求的内容
  3. 如果请求的是静态资源（html），那么直接在网站目录中找到这个资源返回给浏览器

  4. 如果请求的是动态资源（.php）, 那么会现在网站目录中找到这个文件，然后交给对应的模块，去执行这个文件中的后台代码，最终将执行结果返回给浏览器！

    在浏览器地址栏里面输入www.baidu.com之后敲回车发生了什么？

  5. 浏览器接收域名

  6. 发送域名给DNS

  7. DNS返回域名所对应的IP地址

  8. 浏览器向因特网中发出http协议请求

  9. 路由器依据IP地址，把包裹送达IP所对应的服务器

  10. 服务器根据请求报文内容来判断浏览器想要什么，然后处理之后，将数据打包到http协议里面的response响应报文中，然后发送给浏览器。

http协议

概念

Hyper Text Tranfer Protocal

协议：消息传输的格式
作用：用来在浏览器和服务器之间传递数据！ 各个程序开发语言，都可以使用http协议来传输数据

无状态

浏览器在和服务器通信的时候，服务器记不住浏览器，也就是，浏览器对于服务器的每一次请求，服务器都会认为是一个新的人来请求了！！

无状态的问题需要解决

浏览器在访问服务器的时候，服务器会给浏览器颁发一个令牌，令牌就是浏览器身份的标识，下次浏览器再去访问服务器的时候，将这个令牌带给服务器，那么服务器就能够识别这个浏览器的身份了！！

cookie

概念

cookie是一个浏览器中的存储空间 (files协议下是没有cookie的)

cookie中存储的内容的格式为

cookie如何访问

特点

  1. cookie有生命周期的，他是会过期的！！
    过期时间是可以手动设置的，如果不设置过期时间，那么默认就是浏览器关闭，则cookie消失
    过期时间是针对每一个键值对设置的！
  2. cookie是以域名为单位进行存储的，各个域名之间不能共享cookie
  3. cookie的存储容量是有限制的，每个域名下最多存储4KB内容

  4. cookie是分路径存储的，一个域名下的cookie可以分路径进行存储
    子路径中存储cookie，不能再父路径中访问
    父路径中存储的cookie，是可以在子路径中访问的，所以，一般情况下，如果想要在整个域名中共享cookie内容，都会将其存储在根路径中！！

  5. cookie中的信息，会伴随着每一次的请求，发送给服务器，不论什么请求

    • 注意： 一般情况下，不会往cookie中存储大量的数据
  1. cookie不仅可以在浏览器端操作，也可以在服务器端操作
    在服务器端操作其实就是在响应头中添加了一个set-cookie的内容，浏览器接收到这个响应信息之后，会将对应的内容存入cookie中！！

跨域问题

跨域是什么？

由于浏览器的同源策略，url地址中，只要（协议 域名 端口号）任意部分不相同的url地址之间，就会形成跨域，跨域的xhr请求，会被浏览器给终止！！

什么时候会有跨区请求出现

  1. 使用第三方的api的时候
  2. 公司中将web服务器和api服务器隔离开了，各自有自己的服务器！

跨域请求的问题如何解决

浏览器不支持xhr跨域请求！！

  1. jsonp JSON with Padding
    对于服务器端返回的数据有格式的要求（必须是: 回调函数名(数据)）

  2. 修改服务器端的响应头 Access-Control-Allow-Origin:itcast.cn

  3. 反向代理

    将所有的数据请求，全部让web服务器代理，转发给接口服务器，接口服务返回数据给web服务器，web服务器再把数据返回给浏览

    反向代理，会对服务器性能产生影响！

模块化

概念

由于项目功能越来越多，我们不能将所有的代码全部放在一起，这样的话，不利于代码的管理，以及后期维护，和日后的复用！

所以，我们会将代码按照不同的规则将代码拆分成一系列的模块，这个做法就是所谓的模块化！

模块化标准

标准

标准就是为了实现通用的目的，指定的一系列的规则，如果大家都遵守这个规则，那么就可以实现通用的效果

模块化标准

  1. CommonJS: 为后端的js制定的标准，Node.js就是遵守这个CommonJS来实现模块化的！！！

  2. AMD: Async Module Definition(异步模块定义), 实现了AMD标准的模块化技术代表为(require.js) 提倡依赖前置

  3. CMD: Common Module Definition(通用模块定义), 实现了CMD标准的模块化技术代表为(sea.js) 提倡依赖延迟（以来就近） As Lazy As Possible

模块化的实现

  1. 简单以文件作为模块（不推荐）
    将项目中的不同功能的代码，单纯的以文件的形式进行保存

    缺点：

    1. 每个文件中并没有单独的作用域，在文件中声明的变量，都是全局变量，会造成全局变量污染
2. 模块之间的依赖关系，需要通过分析之后，用script标签的顺序来进行确定！

  2. 通过require.js来实现模块化
    ！

require.js

  1. require.js其实就是一个js文件，这个js文件中提供了一些功能，用来帮助我们实现模块化

    – requirejs的原理，实际上它是在页面中动态创建了一个script标签，将要加载的文件的路径放到script的src属性中

  2. 下载requirejs http://requirejs.or

  3. 将requirejs引入页面

  4. 通过requirejs来定义一个模块，使用里面的define函数

  5. 通过requirejs来引用已经创建好的模块，使用里面的require函数

  6. 如果模块有返回值，那么需要用过在require中写一个回调函数，这个回调函数中需要声明一个形参，当模块加载完成之后，模块的返回值会以实参的形式传入require中写的回调函数中的形参，那么回调函数就拿到了模块的返回值了

    注意：有个模块文件中，只允许写一个define函数的调用，如果有多个，那么最终只会执行第一个！

  7. 模块有依赖项的情况

    • 定义模块的时候，如果当前模块要依赖于别的模块的某些功能,那么就需要将别的模块作为当前模块的依赖项。

    • define函数的第一个参数中的数组中，就可以将所有的依赖项的路径（不带.js后缀）放在里面

    • 当定义当前模块的时候，requirejs会先去加载他所有的依赖项再将依赖项的返回值作为参数，传递给当前模块的函数的形参，我们就可以在当前模块中使用其他模块中的功能了

如果模块中有多个依赖项，那么要注意，每个形参和每个依赖项的顺序一定是一一对应的

就算中间有没有返回值的模块，形参也不可以省略,一般会把有返回值的模块的形参写在前面
